## State Management

Three layers, each for a different type of state:

| Layer        | Library     | What it manages                  | Example                               |
| ------------ | ----------- | -------------------------------- | ------------------------------------- |
| Server state | React Query | API data, caching, refetching    | Events list, cluster data             |
| Client state | Zustand     | UI state, ephemeral interactions | Filter drawer open/closed, map bounds |
| URL state    | nuqs        | Shareable, bookmarkable state    | Active filters, selected catalog      |

35 custom React Query hooks in `lib/hooks/` handle all data fetching. Components never call `fetch()` directly.

## Data Flow

```
CSV/Excel upload or scheduled URL fetch
  вҶ’ Payload Workflow queued (ingest queue)
    вҶ’ Dataset detection, schema analysis
      вҶ’ User reviews schema (NEEDS_REVIEW)
        вҶ’ Geocoding via providers (Nominatim вҶ’ Google вҶ’ OpenCage)
          вҶ’ Events created in PostgreSQL
            вҶ’ PostGIS spatial indexing
              вҶ’ React Query cache invalidation
                вҶ’ Map + timeline update
```

## Background Jobs

3-queue architecture using Payload's built-in job system:

| Queue         | Worker               | Purpose                                 |
| ------------- | -------------------- | --------------------------------------- |
| `ingest`      | `worker-ingest`      | File imports, geocoding, event creation |
| `default`     | `worker-default`     | Schedule manager, webhooks              |
| `maintenance` | `worker-maintenance` | Cleanup, cache expiry, audit rotation   |

4 ingest workflows orchestrate the import pipeline. 9 system jobs run on cron schedules. See [Background Jobs](./background-jobs) for the full reference.

## Database

PostgreSQL 17 with PostGIS 3.5 for spatial data. Key patterns:

- **Payload CMS** manages schema and migrations вҖ” all tables in the `payload` schema
- **PostGIS** functions for clustering (`ST_ClusterKMeans`), distance, and spatial queries
- **GeoJSON standard** вҖ” coordinates always in `[longitude, latitude]` order
- **Drizzle ORM** (via Payload) for type-safe database access

## Frontend

- **MapLibre GL JS** for interactive maps with WebGL rendering
- **Server-side clustering** via PostGIS вҖ” the frontend never processes raw event data
- **Apache ECharts** for timeline histograms and charts
- **next-intl** for i18n (English, German) with `localePrefix: "as-needed"` routing
- **Tailwind CSS + shadcn/ui** component library in `packages/ui/`

## Key Design Decisions

**Payload CMS integrated mode** вҖ” CMS runs inside the Next.js process, sharing the same server. No separate API server needed.

**PostGIS over client-side geo** вҖ” All spatial queries, clustering, and distance calculations happen in the database. The frontend receives pre-processed GeoJSON.

**Payload Workflows over state machines** вҖ” The import pipeline uses Payload's native workflow system for task orchestration, replacing the earlier hook-driven state machine.

**React Query over Redux** вҖ” Server state is managed by React Query with intelligent caching and background refetching. Client-only state uses Zustand (lightweight, no boilerplate).

**Monorepo with Turbo** вҖ” Shared packages (`ui`, `assets`, `eslint-config`, etc.) with parallel builds and intelligent caching.
