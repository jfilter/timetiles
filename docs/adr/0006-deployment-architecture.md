# ADR 0006: Deployment Architecture

## Status

Accepted

## Context

TimeTiles is a Next.js application with Payload CMS integrated (not standalone), PostgreSQL with PostGIS for geospatial queries, file-based uploads, and background job processing. It runs as a single Docker container behind Nginx with Let's Encrypt SSL.

## Decision

### Container Architecture

```
Internet → Nginx (SSL termination) → Next.js + Payload CMS → PostgreSQL + PostGIS
                                          ↓
                                    Local filesystem (uploads)
```

**Docker Compose services:**

| Service | Image | Purpose |
|---------|-------|---------|
| web | Multi-stage build (Node 24-alpine) | Next.js app with Payload CMS |
| postgres | kartoza/postgis:17-3.5 | Database with spatial support (ARM64 compatible) |
| nginx | nginx:alpine | Reverse proxy, SSL termination |
| certbot | certbot/certbot | Let's Encrypt certificate renewal |

**Reference:** `deployment/Dockerfile.prod`, `deployment/docker-compose.prod.yml`

### Dockerfile: Multi-Stage Build

| Stage | Purpose |
|-------|---------|
| base | pnpm setup via corepack |
| pruner | `turbo prune web --docker` strips monorepo to web app dependencies |
| deps | Install production dependencies only |
| builder | Next.js build with `SKIP_DB_CHECK=true` (no DB at build time) |
| runner | Minimal runtime: standalone output + static files + uploads dir |

The runner stage runs as `nextjs` user (UID 1001) for security.

### Database: PostgreSQL 17 + PostGIS 3.5

| Setting | Value | Rationale |
|---------|-------|-----------|
| Image | kartoza/postgis | ARM64 support (Apple Silicon dev) |
| Pool (prod) | Default Payload settings | Single-process, no pool contention |
| Pool (test) | max: 5, idleTimeout: 5000ms | Prevents transaction lock issues in parallel tests |
| Custom SQL functions | `cluster_events`, `calculate_event_histogram` | Server-side clustering and histograms |

Migrations are auto-generated by Payload and stored in `apps/web/migrations/` (60+ files). Never create migrations manually.

**Reference:** `lib/config/payload-config-factory.ts`

### File Storage

| Type | Location | Mount |
|------|----------|-------|
| Media uploads | `UPLOAD_DIR/media/` | Bind-mount via `UPLOAD_HOST_DIR` |
| Import files | `UPLOAD_DIR/import-files/` | Same volume |
| Data exports | `DATA_EXPORT_DIR` (default `.exports`) | Same volume |

All file storage is local filesystem. No S3 adapter. This is a single-process constraint (see ADR 0001).

### Background Jobs

Payload CMS's built-in job queue is used for all background processing. Jobs are stored in the `payload_jobs` database table.

| Category | Jobs |
|----------|------|
| Import pipeline | `dataset-detection`, `analyze-duplicates`, `detect-schema`, `validate-schema`, `create-schema-version`, `geocode-batch`, `create-events-batch` |
| Scheduling | `schedule-manager`, `url-fetch`, `cleanup-stuck-scheduled-imports` |
| Maintenance | `cache-cleanup`, `quota-reset`, `schema-maintenance`, `cleanup-approval-locks` |
| User data | `data-export`, `data-export-cleanup`, `execute-account-deletion`, `process-pending-retries` |

Jobs auto-delete on completion. Failed jobs are retained for debugging.

**Reference:** `lib/jobs/import-jobs.ts`, `lib/jobs/handlers/`

### Health Check

The `/api/health` endpoint performs 9 checks:

| Check | What It Validates |
|-------|-------------------|
| Environment | `PAYLOAD_SECRET`, `DATABASE_URL` present |
| Uploads | Directory exists and is writable |
| Geocoding | At least one enabled provider |
| Payload | Can query users collection |
| Migrations | No pending migrations |
| PostGIS | Extension installed |
| SQL functions | `cluster_events`, `calculate_event_histogram` exist |
| Email | SMTP configured or ethereal.email for dev |
| Database size | Reports total DB size |

Returns 200 (healthy/degraded) or 503 (error).

**Reference:** `lib/health.ts`

### Environment Variables

| Variable | Required | Purpose |
|----------|----------|---------|
| `DATABASE_URL` | Yes | PostgreSQL connection string |
| `PAYLOAD_SECRET` | Yes | Payload CMS encryption key |
| `NEXT_PUBLIC_PAYLOAD_URL` | Yes | Public-facing URL (used in emails, CORS) |
| `UPLOAD_DIR` | No | Upload directory (default: `uploads`) |
| `LOG_LEVEL` | No | Logging verbosity (default: `info`) |
| `EMAIL_SMTP_HOST` | No | SMTP server for transactional email |
| `SKIP_DB_CHECK` | Build only | Skip DB connection during Next.js build |

**Reference:** `.env.example`, `deployment/.env.production.example`

### Build-Time Isolation

During `next build`, the app must not connect to a database. This is handled by:
- `SKIP_DB_CHECK=true` in Dockerfile builder stage
- `NEXT_PHASE=phase-production-build` detection in config factory
- Minimal pool settings (`max: 0`) during build
- Migrations skipped, run only at runtime startup

## Consequences

- Single-container deployment is simple but limits horizontal scaling (see ADR 0001)
- Local file storage requires shared filesystem or volume for any future multi-container setup
- kartoza/postgis provides ARM64 support but is a third-party image dependency
- Payload's integrated mode means CMS and app share the same process — no separate CMS service
- Build-time DB isolation adds config complexity but enables CI/CD without database access
- Let's Encrypt via Certbot requires port 80 access for ACME challenges
